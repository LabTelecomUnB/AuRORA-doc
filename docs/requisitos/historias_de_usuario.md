# História de Usuários

O presente artefato tem por objetivo descrever as histórias de usuários associadas aos requisitos elicitados anteriormente, as quais incluem a descrição, critérios de aceitação, regras de negócio, dependências, estimativa e prioridade.

## Metodologia 

As Histórias de Usuário são a base para a descrição de requisitos em metodologias ágeis e tem por objetivo capturar as necessidades reais dos usuários, focar no valor de negócio, facilitar a comunicação entre o stakeholder e as equipes técnicas, permitir a evolução contínua do produto e garantir a testabilidade de cada funcionalidade. 

Elas se iniciam com uma frase concisa que segue o modelo padrão: "Como [tipo de usuário], eu quero [o que ele precisa] para [benefício]". Este formato visa garantir que a funcionalidade esteja ancorada em uma necessidade real do usuário e entregue um valor de negócio perceptível.

Cada história é complementada por uma descrição detalhada, que esclarece as regras de negócio e o comportamento esperado da funcionalidade. As regras devem ser escritas não descrevendo o que deve ser implementado, mas o que deve ser verdadeiro sempre no software. Geralmente são escritas descrevendo condições e consequências e devem ser completamente independentes da interface, ou seja, não descrevem como será implementado mas sim o comportamento esperado. Outro elemento importante é a rastreabilidade. Uma regra de negócio pode — e deve — se relacionar com uma ou várias histórias de usuário e com requisitos. Porém, ela não deve se confundir com nenhum deles. Enquanto uma história de usuário descreve uma necessidade de valor para o usuário, a regra de negócio descreve o mecanismo lógico universal que o sistema deve seguir para que o negócio funcione corretamente.

A definição da história também conta com os Critérios de Aceitação, que são as condições objetivas que precisam ser satisfeitas para que a história seja considerada completa. Nesse contexto, utilizaremos o BDD (Behavior-Driven Development- Desenvolvimento Orientado ao Comportamento) para descrever os critérios de aceitação. Essa metodologia consiste em utilizar uma sintaxe famosa e padronizada, chamada Gherkin, composta pelos termos Given, When e Then (ou, em português, Dado, Quando e Então) para descrever situações específicas de forma estruturada e verificável. Assim, cada história de usuário não fica apenas no nível conceitual, mas é acompanhada de uma descrição precisa do comportamento esperado.

Além disso, 

No contexto ágil, é essencial que a história receba uma Estimativa de Esforço (frequentemente expressa em Pontos de História), a qual indica a complexidade relativa da tarefa. Por fim, a história deve ter uma prioridade atribuída para orientar o sequenciamento do desenvolvimento.
